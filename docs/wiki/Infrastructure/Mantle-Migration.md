## 1. Migration Overview

### Before

The media-downloader project had **manually authored Terraform files** in a `terraform/` directory:

- Each Lambda, API Gateway route, IAM policy, and resource was written by hand
- Adding a new Lambda required editing multiple `.tf` files (lambda resource, API Gateway resource/method/integration, IAM policy, environment variables)
- Environment variables were manually kept in sync between handler code and Terraform
- IAM policies were hand-written and often over-permissioned
- No systematic relationship between TypeScript handler annotations and infrastructure

### After

The project is now **100% generated by `mantle generate infra`**:

- **34 generated `.tf` files** in `infra/` (every file starts with `# This file is generated by Mantle CLI`)
- **0 manual `.tf` files** remaining
- **18 Lambda functions** discovered and wired automatically
- Infrastructure is derived from two sources: `mantle.config.ts` and handler-level `defineLambda()` / `defineApiHandler()` / `defineSqsHandler()` / `defineScheduledHandler()` annotations
- IAM policies are auto-generated from AWS SDK import analysis (`extractServicePermissions()`)
- API Gateway routes are derived from filesystem-based routing (`src/lambdas/api/`)
- Successfully deployed to staging with zero Terraform drift

### Key Stats

| Metric | Value |
|---|---|
| Generated `.tf` files | 34 |
| Manual `.tf` files | 0 |
| Lambda functions | 18 (10 API + 2 SQS + 2 Scheduled + 2 Standalone + 1 S3 + 1 Edge) |
| API routes | 10 endpoints across 4 path prefixes |
| SQS queues | 2 (DownloadQueue, SendPushNotification) |
| EventBridge rules | 1 (DownloadRequested -> DownloadQueue with input_transformer) |
| S3 buckets | 1 (files, with CloudFront + intelligent tiering) |
| DynamoDB tables | 1 (idempotency, with TTL) |
| SNS topics | 2 (push-notifications, operations-alerts) |
| SNS platform apps | 1 (APNS sandbox) |

## 2. mantle.config.ts -- The Single Source of Truth

**File:** `aws-cloudformation-media-downloader/mantle.config.ts`

The entire project infrastructure is configured through a single `defineConfig()` call. Every section maps directly to generated Terraform resources.

### 2.1 Project Name

```typescript
name: 'media-downloader'
```

Sets `var.project_name` default and is used throughout the generated Terraform for naming conventions via `module.core.name_prefix`.

### 2.2 Backend (S3 State)

```typescript
backend: {
  s3: {
    bucket: 'lifegames-media-downloader-tfstate',
    key: 'infra.tfstate',
    region: 'us-west-2',
    encrypt: true,
    dynamodbTable: 'MediaDownloader-TerraformStateLock',
    workspaceKeyPrefix: 'env'
  }
}
```

**Generates:** `backend.tf` -- S3 backend block with DynamoDB state locking. Supports workspace-based environment isolation (`env/staging/infra.tfstate`, `env/prod/infra.tfstate`).

### 2.3 Database (Aurora DSQL)

```typescript
database: { provider: 'aurora-dsql' }
```

**Generates:** `database.tf` -- Sources `mantle/modules/database/aurora-dsql` with name prefix, tags, and deletion protection (enabled in prod). Also generates `generated_dsql_permissions.tf` with per-Lambda DSQL role mappings and `migrate_dsql.tf` for migration automation.

### 2.4 EventBridge (Bus + SQS Targets)

```typescript
eventbridge: {
  bus: 'MediaDownloader',
  sqsTargets: [{
    detailType: 'DownloadRequested',
    queue: 'DownloadQueue',
    inputTransformer: {
      inputPaths: {
        fileId: '$.detail.fileId',
        sourceUrl: '$.detail.sourceUrl',
        correlationId: '$.detail.correlationId',
        userId: '$.detail.userId'
      },
      inputTemplate: `{
  "fileId": <fileId>,
  "sourceUrl": <sourceUrl>,
  "correlationId": <correlationId>,
  "userId": <userId>,
  "attempt": 1
}`
    }
  }]
}
```

**Generates:** `eventbridge.tf` -- EventBridge bus module, `aws_cloudwatch_event_rule` for `DownloadRequested` events, `aws_cloudwatch_event_target` with input transformer that reshapes EventBridge detail into SQS message format, and `aws_sqs_queue_policy` allowing EventBridge to send messages to the queue.

This is the core of the download pipeline: Feedly webhook -> `emitEvent('DownloadRequested')` -> EventBridge -> input_transformer -> SQS DownloadQueue -> StartFileUpload Lambda.

### 2.5 Observability (ADOT + Metrics)

```typescript
observability: {
  adot: true,
  metricsNamespace: 'MediaDownloader'
}
```

**Generates:** ADOT layer ARNs in `locals.tf` (both arm64 and x86_64 variants), `OPENTELEMETRY_EXTENSION_LOG_LEVEL` and `OPENTELEMETRY_COLLECTOR_CONFIG_URI` in `common_lambda_env`, and `METRICS_NAMESPACE` set to `MediaDownloader`. Every Lambda (except edge functions) gets the ADOT layer attached.

### 2.6 Secrets (SOPS)

```typescript
secrets: {
  provider: 'sops',
  filePattern: 'secrets.{env}.enc.yaml'
}
```

**Generates:** `sops.tf` -- `data "sops_file" "secrets"` with environment-aware file path (`secrets.staging.enc.yaml` or `secrets.prod.enc.yaml`). Also adds the `carlpett/sops` provider to `main.tf`'s `required_providers`. Handler-level `secrets: { VAR_NAME: 'sops.key.path' }` annotations are resolved to `data.sops_file.secrets.data["sops.key.path"]` in Lambda env vars.

### 2.7 SNS (Topics + Platform Applications)

```typescript
sns: {
  topics: [
    { name: 'push-notifications' },
    { name: 'operations-alerts' }
  ],
  platformApplications: [{
    name: 'media-downloader',
    platform: 'APNS_SANDBOX',
    credentialSecret: 'apns.staging.privateKey',
    principalSecret: 'apns.staging.certificate',
    resourceName: 'apns'
  }]
}
```

**Generates:** `sns.tf` -- Two `aws_sns_topic` resources (`PushNotifications`, `OperationsAlerts`), SNS logging IAM role/policy, and `aws_sns_platform_application.apns` (note: `resourceName: 'apns'` overrides the default TF resource name). Platform credentials reference SOPS secrets. Environment-aware platform switching (`APNS_SANDBOX` for non-prod, `APNS` for production).

### 2.8 DynamoDB (with tableNameOverride)

```typescript
dynamodb: [{
  name: 'idempotency',
  tableNameOverride: '${module.core.name_prefix}-MediaDownloader-Idempotency',
  hashKey: 'id',
  attributes: [{ name: 'id', type: 'S' }],
  ttlAttribute: 'expiration'
}]
```

**Generates:** `dynamodb_idempotency.tf` -- Sources `mantle/modules/dynamodb` with the exact legacy table name via `table_name_override`. This preserves the existing DynamoDB table name so no data migration is needed. TTL is configured on the `expiration` attribute.

### 2.9 Storage (with bucketNameOverride, CloudFront, Intelligent Tiering, Assets)

```typescript
storage: [{
  name: 'files',
  bucketNameOverride: 'lifegames-${var.resource_prefix}-media-files-${module.core.account_id}',
  cloudfront: true,
  intelligentTiering: true,
  assets: ['videos/default-file.mp4']
}]
```

**Generates:** `storage_files.tf` -- Sources `mantle/modules/storage` with `bucket_name_override` (preserves legacy bucket naming), CloudFront distribution enabled, and S3 Intelligent-Tiering. The `assets` array generates an `aws_s3_object` resource that uploads `static/videos/default-file.mp4` to the bucket with content-type detection and etag-based change tracking.

### 2.10 Queues (with envAlias)

```typescript
queues: [
  { name: 'DownloadQueue' },
  { name: 'SendPushNotification', envAlias: 'SNS_QUEUE_URL' }
]
```

**Generates:** `queue_DownloadQueue.tf` and `queue_SendPushNotification.tf` -- Each sources `mantle/modules/queue`. The `envAlias: 'SNS_QUEUE_URL'` on SendPushNotification means that when a Lambda references `SNS_QUEUE_URL` in its environment variables, the generator resolves it to `module.queue_SendPushNotification.queue_url` instead of expecting a variable named `SNS_QUEUE_URL`.

### 2.11 CloudFront (API Distribution for Lambda@Edge)

```typescript
cloudfront: {
  apiDistribution: {
    geoRestriction: { type: 'whitelist', locations: ['US'] },
    forwardedHeaders: ['X-API-Key', 'Authorization', 'User-Agent'],
    cacheTtl: { default: 0, min: 0, max: 0 }
  }
}
```

**Generates:** CloudFront distribution in `lambda_cloudfront_middleware.tf` -- Origin points to API Gateway, Lambda@Edge function association for `origin-request`, geo-restriction to US only, header forwarding for auth, and zero-TTL caching (pass-through to origin).

### 2.12 Custom Variables (9 Project-Specific TF Variables)

```typescript
customVariables: [
  { name: 'resource_prefix', type: 'string', description: '...', validation: {...} },
  { name: 'download_reserved_concurrency', type: 'number', default: '10', ... },
  { name: 'api_throttle_burst_limit', type: 'number', default: '100', ... },
  { name: 'api_throttle_rate_limit', type: 'number', default: '50', ... },
  { name: 'api_quota_limit', type: 'number', default: '10000', ... },
  { name: 'dsql_deletion_protection', type: 'bool', default: 'true', ... },
  { name: 'enable_cloudwatch_dashboard', type: 'bool', default: 'false', ... },
  { name: 'enable_cloudwatch_alarms', type: 'bool', default: 'false', ... },
  { name: 'cors_allowed_origins', type: 'list(string)', default: '[]', validation: {...} }
]
```

**Generates:** Custom variable blocks appended to `variables.tf` with types, defaults, descriptions, and validation rules. These are project-specific variables that don't fit into standard Mantle configuration -- things like resource naming prefixes, API throttle limits, and feature flags.

## 3. Handler Annotations -- How Code Drives Infrastructure

Every Lambda handler file contains annotations that `mantle generate infra` reads via ts-morph AST analysis. These annotations drive the generated Terraform.

### 3.1 Annotation Reference

| Annotation | Effect on Generated TF |
|---|---|
| `defineApiHandler({ auth: 'authorizer' })` | `authorization = "CUSTOM"` + `authorizer_id` on API Gateway method |
| `defineApiHandler({ auth: 'none' })` | `authorization = "NONE"` on API Gateway method |
| `defineLambda({ secrets: { VAR: 'sops.path' } })` | `VAR = data.sops_file.secrets.data["sops.path"]` in env vars |
| `defineLambda({ staticEnvVars: { VAR: 'value' } })` | `VAR = "value"` hardcoded in env vars |
| `defineLambda({ architecture: 'x86_64' })` | `architecture = "x86_64"` + x86_64 ADOT layer |
| `defineLambda({ edge: true })` | `is_edge_function = true`, `providers = { aws = aws.us_east_1 }`, `lambda_edge_assume_role_policy`, no ADOT layer |
| `defineSqsHandler({ queue: 'QueueName' })` | `sqs_trigger_arn` + `sqs_trigger_enabled = true` |
| `defineLambda({ reservedConcurrency: N })` | `reserved_concurrent_executions = N` |
| `defineLambda({ timeout: N })` | `timeout = N` |
| `defineLambda({ memorySize: N })` | `memory_size = N` |
| `defineScheduledHandler({ schedule: { expression: '...' } })` | `schedule_expression = "..."` on Lambda module |
| `defineLambda({ staticAssets: ['path'] })` | `ASSET_*` env vars pointing to S3 object + CloudFront URL |

## 5. Generator Architecture

The flow when running `mantle generate infra`:

```
mantle generate infra
  |
  v
loadProjectConfig()          -- Reads mantle.config.ts via esbuild transpilation
  |                             Returns typed MantleConfig object
  v
discoverLambdas()            -- Scans src/lambdas/ via ts-morph AST analysis
  |                             Finds defineLambda(), defineApiHandler(), defineSqsHandler(),
  |                             defineScheduledHandler(), defineS3Handler(), defineAuthorizerHandler()
  |                             Extracts: secrets, staticEnvVars, architecture, edge, timeout,
  |                             memorySize, reservedConcurrency, queue, schedule, auth, staticAssets
  |                             Returns DiscoveredLambda[] with type: 'api' | 'standalone' | 'eventbridge' | 'sqs' | 'scheduled' | 's3'
  v
extractServicePermissions()  -- Scans handler source files for AWS SDK imports
  |                             Detects: @mantleframework/aws imports (sendMessage, publish,
  |                             headObject, createPlatformEndpoint, listSubscriptionsByTopic, etc.)
  |                             Maps imports to IAM actions (sqs:SendMessage, sns:Publish, s3:HeadObject, etc.)
  |                             Returns Record<string, LambdaServicePermissions>
  v
generate*Tf() functions      -- Each function produces one or more .tf files:
  |
  |  generateMainTf()        -- terraform block, providers, core module, api-gateway module
  |  generateBackendTf()     -- S3 backend from config.backend
  |  generateVariablesTf()   -- Standard + env-var-derived + custom variables
  |  generateLocalsTf()      -- ADOT layers, common env, EventBridge bus name
  |  generateOutputsTf()     -- All module outputs
  |  generateLambdaTf()      -- Per-Lambda: module block, API Gateway resources, env vars, IAM policies
  |  generateApiRoutesTf()   -- Shared API Gateway path resources
  |  generateEventBridgeTf() -- Bus module, rules, targets, queue policies
  |  generateSopsTf()        -- SOPS data source
  |  generateSnsTf()         -- Topics, platform apps, logging IAM
  |  generateStorageTf()     -- S3 + CloudFront modules, asset uploads
  |  generateQueueTf()       -- SQS queue modules
  |  generateDynamoDbTf()    -- DynamoDB table modules
  |  generateDatabaseTf()    -- Aurora DSQL module
  |  generateMigrateDsqlTf() -- Migration Lambda + invocation
  |  generatePermissionsTf() -- DSQL role mappings
  |
  v
safeWriteGeneratedFile()     -- Checks first line of existing file for generated header
  |                             If header present or file doesn't exist: write new content
  |                             If header absent (file was "ejected"): skip, preserve manual edits
  v
resolveEnvVar()              -- Maps environment variable names to Terraform expressions:
  |                             BUCKET          -> module.storage_*.bucket_id
  |                             CLOUDFRONT_DOMAIN -> module.storage_*.cloudfront_domain_name
  |                             *_QUEUE_URL     -> module.queue_*.queue_url (respects envAlias)
  |                             *_TOPIC_ARN     -> aws_sns_topic.*.arn
  |                             PLATFORM_APPLICATION_ARN -> aws_sns_platform_application.*.arn
  |                             IDEMPOTENCY_TABLE_NAME -> module.dynamodb_*.table_name
  |                             AUTH_BASE_URL   -> API Gateway URL construction
  |                             DSQL_ROLE_NAME  -> local.lambda_dsql_roles[name].role_name
  |                             secrets keys    -> data.sops_file.secrets.data["key"]
  |                             staticEnvVars   -> literal string values
  v
buildInlinePolicies()        -- Generates IAM inline policies from:
  |                             - extractServicePermissions() results (SQS, SNS, S3, API Gateway, DynamoDB)
  |                             - Environment variable patterns (queue -> sqs:SendMessage, etc.)
  |                             Produces minimal-privilege policies per Lambda
  v
Output: 34 .tf files in infra/
```

### Key Implementation Details

- **ts-morph AST analysis** parses TypeScript without executing it. It finds `defineLambda()`, `defineApiHandler()`, etc. call expressions and extracts their configuration objects.
- **Ejection mechanism**: Removing the `# This file is generated by Mantle CLI` header from any file "ejects" it. Future `mantle generate infra` runs skip ejected files, allowing manual customization while keeping everything else generated.
- **Conflict detection**: The generator checks for ejected storage and queue files before generating new ones, preventing duplicate resource definitions.
- **Environment variable resolution** (`resolveEnvVar`) is the bridge between handler code and Terraform. When a handler uses `getRequiredEnv('BUCKET')`, the generator knows to wire it to `module.storage_files.bucket_id`.

## 6. Key Patterns

### 6.1 Lambda Environment Variables: merge(local.common_lambda_env, {...})

Every Lambda (except edge functions and MigrateDSQL) uses:

```hcl
environment_variables = merge(local.common_lambda_env, {
    # Lambda-specific variables
    API_BEARER_TOKEN = var.api_bearer_token
    DSQL_ROLE_NAME   = local.lambda_dsql_roles["LambdaName"].role_name
    # ... additional per-handler vars
})
```

`local.common_lambda_env` provides:
- `OPENTELEMETRY_EXTENSION_LOG_LEVEL` -- ADOT log level
- `OPENTELEMETRY_COLLECTOR_CONFIG_URI` -- ADOT collector config path
- `NODE_OPTIONS` -- `--no-deprecation`
- `LOG_LEVEL` -- from `var.log_level`
- `ENVIRONMENT` -- from `var.environment`
- `DSQL_ENDPOINT` -- Aurora DSQL cluster endpoint
- `DSQL_REGION` -- AWS region
- `METRICS_NAMESPACE` -- CloudWatch metrics namespace

### 6.2 locals.tf with ADOT Layer ARNs

```hcl
locals {
  adot_layer_arn        = "arn:aws:lambda:${module.core.region}:901920570463:layer:aws-otel-nodejs-arm64-ver-1-30-2:1"
  adot_layer_arn_x86_64 = "arn:aws:lambda:${module.core.region}:901920570463:layer:aws-otel-nodejs-amd64-ver-1-30-2:1"
}
```

Most Lambdas use `layers = [local.adot_layer_arn]` (arm64). StartFileUpload uses `layers = [local.adot_layer_arn_x86_64]` because it has `architecture: 'x86_64'` for yt-dlp binary compatibility. Edge functions get no layers.

### 6.3 bucket_name_override / table_name_override for Legacy Naming

The media-downloader predates Mantle's naming conventions. To avoid destroying and recreating existing AWS resources:

```hcl
# Storage
bucket_name_override = "lifegames-${var.resource_prefix}-media-files-${module.core.account_id}"

# DynamoDB
table_name_override = "${module.core.name_prefix}-MediaDownloader-Idempotency"
```

These override the module-default names while keeping the same Terraform module interface.

### 6.4 resourceName on SNS Platform Applications

```typescript
platformApplications: [{
  name: 'media-downloader',
  resourceName: 'apns',  // Overrides TF resource identifier
  // ...
}]
```

Without `resourceName`, the TF resource would be `aws_sns_platform_application.media_downloader`. With it, the resource is `aws_sns_platform_application.apns`, which matches the original hand-written Terraform resource name and avoids a destroy/recreate cycle.

### 6.5 envAlias on Queues for Naming Mismatches

```typescript
queues: [
  { name: 'SendPushNotification', envAlias: 'SNS_QUEUE_URL' }
]
```

The handlers use `getRequiredEnv('SNS_QUEUE_URL')` (a legacy name) but the queue is named `SendPushNotification`. The `envAlias` tells `resolveEnvVar()` that `SNS_QUEUE_URL` should map to `module.queue_SendPushNotification.queue_url`.

### 6.6 moved {} Blocks (Now Removed)

During the cutover, `moved {}` blocks were used in a temporary `moved.tf` file to migrate Terraform state from raw resources to module-wrapped resources without destroying infrastructure:

```hcl
# Example (no longer present):
moved {
  from = aws_lambda_function.StartFileUpload
  to   = module.lambda_start_file_upload.aws_lambda_function.this
}
```

These were removed in commit `5ed20941` after the state migration was complete and verified.

## 9. Known Limitations and Follow-ups

### 9.1 api_bearer_token Variable

The `api_bearer_token` variable is regenerated on each `mantle generate infra` run without a `default = ""`. This was manually fixed by adding `default = ""` to avoid Terraform prompting for a value. The generator should be updated to include the default.

**Current state:** Fixed in the generated output (default present in `variables.tf`).

### 9.2 Unused TF Variables for Secrets/Static Env Vars

Variables are generated for environment variable names that are actually handled by SOPS secrets or `staticEnvVars`. For example, `var.apns_host` exists in `variables.tf` even though PruneDevices gets `APNS_HOST` from `staticEnvVars: { APNS_HOST: 'api.sandbox.push.apple.com' }`.

**Impact:** Harmless -- these variables have `default = ""` and are never referenced. They add noise to `variables.tf` but do not affect functionality.

### 9.3 Removed data "http" "icanhazip"

The original Terraform had a convenience data source:

```hcl
data "http" "icanhazip" {
  url = "https://icanhazip.com"
}
```

This was used for debugging to capture the deploy-time public IP. It is not generated by Mantle and was removed during the migration.

### 9.4 Output Name Changes

Some output names changed from the original hand-written Terraform:

| Original | Generated |
|---|---|
| `media_bucket_name` | `storage_files_bucket_arn` |
| `media_bucket_arn` | `storage_files_bucket_arn` |
| `cloudfront_distribution_domain` | `storage_files_cloudfront_domain` |

Any external systems referencing these outputs by name will need updating.

### 9.5 FileHelpers Lambda

The file `src/lambdas/sqs/StartFileUpload/file-helpers.ts` is discovered as a separate Lambda (`FileHelpers`) because it lives in the `src/lambdas/` directory tree. It generates `lambda_file_helpers.tf` with a full Lambda module. In practice, this file is a utility module imported by StartFileUpload and should not be a standalone Lambda. This is a minor discovery heuristic issue -- the generated Lambda is harmless but unnecessary.
